# Cloud Computing

## Cloud Computing - Introduction
> Delivery of computing services over internet.
* azure cloud includes computing, storage and network services.
* It also includes IOT, Machine learning and AI

## Shared Responsibility
>Divides tasks between cloud provider and user.
* __on-prem__ user responsible of everything
* __cloud__ shared responsibility

### always user's responsibility
* Data security
* Devices connected
* User accounts

### always provider's responsibility
* Physical security
* Network security
* Infrastructure (varies)

### shared responsibility
* Operating systems
* Network controls
* Applications
* Identity and infrastructure

![shared-responsibility](https://learn.microsoft.com/en-us/training/wwl-azure/describe-cloud-compute/media/shared-responsibility-b3829bfe.svg)

### Service Type Responsibility

* __IaaS__: Most on user (like renting space)
* __PaaS__: Shared responsibility
* __SaaS__: Least on user (like using online software)

## Cloud Models
four types of cloud models for deploying cloud resources:
1. __Private Cloud:__ Like a corporate datacenter, but accessible over the internet. Offers high control but can be expensive.
2. __Public Cloud:__ Resources are shared and delivered by a third-party provider. Offers flexibility and lower cost, but less control.
3. __Hybrid Cloud:__ Combines public and private clouds for flexibility, security, and workload management.
4. __Multi-Cloud:__ Uses multiple public cloud providers for specific features or migration purposes.

### cloud models comparison
| Feature | Public Cloud | Private Cloud | Hybrid Cloud | Multi-Cloud |
|---|---|---|---|---|
| Cost | Pay-per-use | High upfront cost | Varies | Varies |
| Control | Least control | Most control | Shared control | Varies |
| Security | Shared responsibility | Highest security | Varies | Varies |
| Scalability | Easy to scale | Requires hardware investment | Flexible scaling | Varies |

### cloud technologies
1. __azure arc:__ set of azure technologies to manage cloud environments across public, private, hybrid, and multi-cloud scenarios.
2. __azure vmware:__ runs existing VMware workloads seamlessly in Azure for migration or cloud adoption.

## Consumption Based Models
1. __CapEx__  
    * Capital expenditure
    * one time purchase of infrastructure
    * Inflexible and potentially wasteful
2. __OpEx__ 
    * operational expenditure
    * Pay as you go
    * subscription based infrastructure
>Cloud computing is _OpEx_ model

# Cloud Benefits
* High availability and scalability
* Reliability and predictability
* Security and governance
* Manageability

## High Availabilty
* Ensures critical applications are always available when needed.
* Cloud providers offer uptime guarantees (SLAs) for their services.

## Scalability
* Allows you to adjust resources to meet changing demands.
* Pay-as-you-go model: only pay for what you use.

### types of scalability
1. __Vertical scaling:__ Increase/decrease capabilities of a single resource (e.g., CPU, RAM).
2. __Horizontal scaling:__ Add/remove resources (e.g., VMs, containers).

## Reliability
* Cloud's decentralized design ensures redundancy - if one region fails, others stay operational.
* Applications can automatically leverage this for high uptime.

## Predictability
* __Performance Predictability:__ Cloud features like autoscaling and load balancing help maintain consistent performance.
* __Cost Predictability:__ Cloud offers real-time usage tracking, resource optimization, and cost analytics for better budgeting.

## Security and Governance
* __Standardized Deployments:__ Templates ensure resources meet company standards and regulations.
* __Easy Updates:__ Update all deployments to new standards as needed.
* __Auditing:__ Cloud-based auditing identifies non-compliant resources and suggests fixes.
* __Security Options:__ Choose the cloud service (IaaS, PaaS, SaaS) that best meets your security needs.
* __Automatic Patching:__ Some services automatically apply security updates for better security and compliance.
* __DDoS Protection:__ Cloud providers are well-equipped to handle DDoS attacks, improving network security.

## Manageability
There are two main aspects of cloud manageability:

* **Management of the cloud:** This refers to managing your cloud resources. You can:
    * Automatically scale resources based on needs.
    * Deploy resources from pre-configured templates.
    * Monitor resource health and automate replacements.
    * Receive real-time performance alerts.
* **Management in the cloud:** This refers to how you manage your cloud environment. You can use:
    * Web portal
    * Command line interface (CLI)
    * APIs
    * PowerShell

* __Development framework:__ Build and customize cloud applications using pre-built components.

    * Data within the application

# Core Architectural Components

## Azure physical infrastructure
> Data centers -> Availabilty zones -> Regions -> Region pairs

* __data centers:__ Big warehouses filled with servers and storage devices, all connected by high-speed internet.
* __availability zones:__ Physically separated nearby datacenters for extra reliability
* __regions:__ Group of availability zones in a particular area
* __region-pair:__ Pair of nearby regions to prevent outage due to disasters
* __geography:__ region pairs exists in a geography
* __Sovereign Regions:__ Instances of Azure that are isolated from the main instance of Azure

![physical-infra](https://learn.microsoft.com/en-us/training/wwl-azure/describe-core-architectural-components-of-azure/media/region-pairs-7c495a33.png)

## Azure management infrastructure
> Accounts -> Subscription -> Resource Groups -> Resources

### resources
* basic building block of Azure
* Anything you created, provisioned, deployed, etc. is a resource
* Eg - Virtual Machines (VMs), virtual networks, databases, cognitive services

### resource group
* Resource groups are simply groupings of resources
* When you create a resource, you’re required to place it into a resource group
* resource groups can't be nested
* one resource can be only on one group

### subscription
* Unit of management, billing, and scale
* An Azure subscription links to an Azure account, which is an identity in Microsoft Entra ID or in a directory that Microsoft Entra ID trusts.

### management groups _(optional)_
* Azure management groups provide a level of scope above subscriptions.
* You organize subscriptions into containers called management groups and apply governance conditions to the management groups.

![image](https://learn.microsoft.com/en-us/training/wwl-azure/describe-core-architectural-components-of-azure/media/management-groups-subscriptions-dfd5a108.png)

### Azure account
* __Azure account__ provide access to azure 
* __Subscription__ Manages your resource
* Azure account is required to create subscriptions
* Subscriptions are required to create resources
* Every resources must be in a resource group

![azure-account](https://learn.microsoft.com/en-us/training/wwl-azure/describe-core-architectural-components-of-azure/media/account-scope-levels-9ceb3abd.png)

### kubernetes vs docker
* __docker__
    * tool to build and manage individual container
    * Creates container images
    * Run, stop and manage container
    * __Doker Hub__ is public registry community where developers share images and resuse existing images.
* __Kubernetes__
     * deployment, scaling, and management of multiple containers
     * Load Balancing when same app is deployed in multiple containers
     * __Orchestration:__ manages the lifecycle of your containers, ensuring they run healthy and efficiently. This includes tasks like health checks, restarting failed containers, and scheduling them across available resources.

* Docker focuses on individual containers, providing tools for building, running, and managing them.
* Kubernetes focuses on managing containerized applications at scale, orchestrating deployments, scaling, and ensuring their overall health.
* Docker and Kubernetes can work together seamlessly. You can use Docker to build your container images and then leverage Kubernetes to deploy and manage them in a production environment.

## Azure virtual networking
> enables resource communication with each other, users and on-prem

### architecture of azure virtual networking
* __architecture__
    * Resource Group can have one or multiple VNets
    * Vnets can have multiple subnets
    * subnets have azure resources
    * azure resource have private IP (default & mandatory) and public IP(optional)
* __Virtual networks (Vnets)__
    * vNets have own IP address pool
    * two resources in different VNets can have same private IP but not same public IP
    * Resources from one vNet can't communicate with resource from another VNets using private IP
    * VNets can communicate each other using two ways
        1. Virtual Network Peering - within azure cloud
        2. VPN Gateway (Site-to-Site) - on-prem to azure cloud
* __subnets__
    * Segements VNets in isolated containers of resource.

### azure virtual networking capabilities
* __Isolation and segmentation:__
    * Multiple VNets
    * Divide VNets IP range within subnets
    * for name resolution, azure or external DNS can be confugured with VNets
* __Internet communications__
    * To make resource accessible over internet public IP is assigned to resource with public load balancer.
* __Communicate between Azure resources__
    * __Virtual network peering:__ VNets to VNets
    * __Azure Service Endpoints__ VNets to Azure service outside VNet
* __Communicate with on-premises resources__
    * __Point to Site (P2S) VPN:__ remote user to VNets
    * __Site to Site (P2S) VPN:__ Entire on-prem to VNets
    * __ExpressRoute:__ Dedicated connection between on-prem and azure without internet
* __Route network traffic__
    * __Default routing__
        * Between subnets within VNets
        * Between multiple VNets _(if VNet peering configured)_
        * on-prem network _(if S2S vpn configured)_
        * Internet _(resource with public IP address)_
    * __customized routing__
        * Route table is used for customized routing
        * Route tables are sets of rules that define how network traffic is directed within a VNet.
* __Filter network traffic__
    * filter traffic between subnets within VNets
    * __Network Security Groups (NSGs):__
        * NSGs are attached to either specific subnet or entire VNets
        * Rules are defined to control network traffic
        * Rules either `Allow` or `Deny` traffic
        * Act like firewall controlling incoming and outgoing network traffic
    * __Network Virtual Appliances (NVAs):__
        * Specialized VMs created and deployed within VNet
        * NVA perform dedicated network functions
        * Network functions
            * Firewalls
            * WAN Optimization
            * Load Balancing
* Connect virtual networks


* Supports both public and private endpoints to enable communication between external or internal resources with other internal resources.

### Azure VPN
> connects two trusted network over untrusted network (internet)

* __Azure VPN Gateways__
    * Secure connection between on-prem to azure or multiple VNets
    * Site-to-Site (S2S), Point-to-Site (P2S), Network-to-Network (N2N) work same as VNet peering with some differences
    * VPN Gateways instance are deployed in dedicated subnet in a VNet
    * Each VNet can have only one VPN gateways
    * Types of VPN Gateways
        * __Policy-based:__ Gateway have predefined list of IP Address and incoming  pakcets with matched IP are allowed.
        * __Route-based (Preferred):__ uses route table which include all subnets dynamically.
    * High availabilty options
        * __Active/Standby (Default):__ Deploys two VPN gateway instances (one active, one standby). If the active fails, the standby takes over automatically with minimal disruption.
        * __Active/Active (with BGP):__ Uses two VPN gateways, each with a unique public IP. On-premises devices create separate tunnels to each IP, offering higher availability.
        * __ExpressRoute Failover:__ Uses a VPN gateway as backup for ExpressRoute connections in case of outages.
        * __Zone-redundant Gateways (Availability Zones):__ Physically separates gateways within a region for increased resiliency against zone-level failures. Requires specific SKUs and public IP types.

### azure expressroute
> dedicated connection between on-prem and azure cloud

* Benefits
    * Global Reach: Connect to Microsoft cloud services in all regions within a specific geopolitical area. Additionally, ExpressRoute Global Reach allows data exchange across your geographically dispersed facilities.
    * Dynamic Routing: Border Gateway Protocol (BGP) facilitates automatic route exchange between your network and Microsoft services.
    * Built-in Redundancy: Every ExpressRoute peering location has built-in redundancy for high availability, and you can configure multiple circuits for added security.
    * Access to Various Services: Directly connect to Azure services like Virtual Machines, Cosmos DB, Storage, and more, along with Microsoft 365 and Dynamics 365.
    
* __Connectivity Models:__
    * __CloudExchange Colocation:__ If your facility is co-located at a cloud exchange provider (like an ISP), you can request a virtual cross-connection to Microsoft's network.
    * __Point-to-Point Ethernet Connection:__ Establish a dedicated point-to-point connection between your facility and Microsoft's network.
    * __Any-to-Any Connection:__ Integrate your Wide Area Network (WAN) with Azure, allowing connections from your offices and datacenters.
    * __ExpressRoute Direct:__ Connect directly into Microsoft's global network at peering locations for high-bandwidth (100 Gbps or 10 Gbps) and Active/Active configurations.

## azure storage redudancy
Azure Storage always stores multiple copies of your data so that it's protected from planned and unplanned events

1. Locally redundant storage (LRS) - replicates your data three times within a single data center in the primary region
2. Zone-redundant storage (ZRS) - replicates your Azure Storage data synchronously across three Azure availability zones in the primary region
3. Geo-redundant storage (GRS) - copies your data synchronously three times within a single physical location in the primary region using LRS. It then copies your data asynchronously to a single physical location in the secondary region (the region pair) using LRS

## Types of storage services
1. __Blobs__: A massively scalable object store for text and binary data. Also includes support for big data analytics through Data Lake Storage Gen2
2. __Files__: Managed file shares for cloud or on-premises deployments
3. __Queues__: A messaging store for reliable messaging between application components.
4. __Disks__: Block-level storage volumes for Azure VMs
5. __Tables__: NoSQL table option for structured, non-relational data

## azure blobs
* can store massive unstructured amounts of data, such as text or binary data.
* can manage thousands of simultaneous uploads, massive amounts of video data, constantly growing log files, and can be reached from anywhere with an internet connection.
* Blob is suitable for
    * Serving images or documents directly to a browser.
    * Storing files for distributed access.
    * Streaming video and audio.
    * Storing data for backup and restore, disaster recovery, and archiving.
    * Storing data for analysis by an on-premises or Azure-hosted service.

### Blob storage tiers

1. __Hot access tier__
    * __Purpose__: For data that you access frequently.
    * __Performance__: High-performance, optimized for fast read and write operations.
    * __Cost__: Higher storage cost but lower access cost.
    * __Use Cases__: Active data, such as websites, databases, and applications that need quick access to data.
2. __Cool access tier__
    * __Purpose__: For data that you access infrequently but still need to retrieve occasionally.
    * __Performance__: Slightly lower performance compared to the hot tier, but still suitable for data that is accessed less often.
    * __Cost__: Lower storage cost but higher access cost compared to the hot tier.
    * __Use Cases__: Backup data, older files that aren’t accessed daily but still need to be available when required.

4. __Archived access tier__ 
    * __Purpose__: For data that you rarely access and can tolerate retrieval delays.
    * __Performance__: Very low performance, with data retrieval taking hours.
    * __Cost__: Lowest storage cost but highest access cost.
    * __Use Cases__: Long-term storage, such as archival data, compliance records, or any data that you need to keep for historical purposes but don't need to access frequently.


## azure files

## azure queues

* Queues are commonly used to create a backlog of work to process asynchronously.
* Queue storage can be combined with compute functions like Azure Functions to take an action when a message is received.

## azure disk
* block-level storage volumes managed by Azure for use with Azure VMs

## azure tables
* stores large amounts of structured data.
* Azure tables are a NoSQL datastore that accepts authenticated calls from inside and outside the Azure cloud

## Azure storage tools
### AzCopy
* AzCopy is a command-line utility that you can use to copy blobs or files to or from your storage account. 
* With AzCopy, you can upload files, download files, copy files between storage accounts, and even synchronize files.

### Azure Storage Explorer
* It is a standalone app that provides a graphical interface to manage files and blobs in your Azure Storage Account
* It works on Windows, macOS, and Linux operating systems.
* It uses AzCopy on the backend to perform all of the file and blob management tasks.

### Azure File Sync
* Azure File Sync maintains a bidirectional synchronization of files between your on-premises and cloud Windows servers.
* Tool that lets you centralize your file shares in Azure Files and keep the flexibility, performance, and compatibility of a Windows file server.
* Turning your Windows file server into a miniature content delivery network.

# Azure identity, access and security

Direcotry Services - Identity & Access management

## active directory
* service developed by microsoft for windows domain networks.
* manage and organize network resources, such as users, computers, and other devices
* stores information about objects on a network and makes this information available to administrators and users.
* helps in managing permissions and access to networked resources.

Key components of AD are
### Domains
* Collection of objects that shares same AD database
* Provides a boundary for security and administration.

### objects
* These are entities in the AD database.
* Common objects include:
    * __Users__: Accounts representing people.
    * __Groups__: Collections of users or other objects.
    * __Computers__: Devices on the network.
    * __Printers__: Network printers.

### Organizational Units (OUs)
* Containers within a domain that can hold objects.
* Help organize and manage objects more easily by grouping them logically.

### domain controllers
* It is a physical machine or server that hosts AD services
* It responds to security authentication requests (like logging in to a computer) and security checks.

Use of active directory
* Authentication
* Authorization
* Centralized Management
* Group Policy
* Replication
* SSO feature - Single sign in for all application access

## azure directory services
Active Directory - Directory service for on-prem environment
Azure Entra ID - directory service on azure cloud environment.

### AAD uses
* Authentication
    * verify identity to access applications and resources
    * providing functionality such as SSPR, MFA, a custom list of banned passwords, and smart lockout services.
    * Authenticator can use three factor 
        * What I know
        * What I have
        * Who I am
* autherization
    * What can I access

* Single sign-on
    * one account to access all applications
* Application management

* Device management

### Microsoft Entra Connect
* It is used to connect on-prem ad to entra id
* synchronizes user identities between on-premises Active Directory and Microsoft Entra ID
*  Microsoft Entra Connect synchronizes changes between both identity systems, so you can use features like SSO, multifactor authentication, and self-service password reset under both systems.

### Microsoft Entra Domain Services
* It is similar to onprem Domain controllers

#### working of Microsoft Entra Domain Services
* Unique namespace is created with the name of domain while creating Microsoft Entra Domain Services.
* Two Windows Server domain controllers are then deployed into your selected Azure region

## azure policy
* Azure Policy is used to control or audit resources

### Keypoints
* __Define Policies__: Create individual policies or group them into _"initiatives"_ for easier management.
* __Enforce Compliance__: Azure Policy checks resources for compliance with your policies and can prevent non-compliant deployments.
* __Hierarchical Enforcement__: Set policies at different levels (resource, group, subscription) with automatic inheritance for nested structures.
* __Built-in Definitions__: Leverage pre-built policies for common services (Storage, Networking, etc.)
* __Continuous Evaluation__: Azure Policy evaluates existing and new resources for compliance.
* __Automated Remediation (Optional)__: Policies can fix non-compliant resources automatically (e.g., adding missing tags).
* __Exception Handling__: Flag specific resources to prevent automatic remediation.
* __Azure DevOps Integration__: Integrate policy enforcement into your CI/CD pipelines.

## resource locks
* Resource locks prevent accidental deletion or modification of your Azure resources, even with Azure RBAC permissions in place.

![Azure resource locks](https://learn.microsoft.com/en-us/training/wwl-azure/describe-features-tools-azure-for-governance-compliance/media/resource-lock-54695e43.png)

### Application:
* Can be applied to individual resources, resource groups, or entire subscriptions.
* Inherited by child resources within a locked resource group.

### Types of Resource Locks
* __Delete Lock:__ Allows authorized users to read and modify a resource, but not delete it.
* __Read-Only Lock:__ Similar to restricting access to the Reader role: users can only view the resource, no modifications or deletion allowed.

![Types of resource lock](https://learn.microsoft.com/en-us/training/wwl-azure/describe-features-tools-azure-for-governance-compliance/media/read-only-lock-e7777623.png)

### Managing Resource Locks
* Managed through Azure portal, PowerShell, Azure CLI, or Azure Resource Manager templates.
* To view, add, or delete locks in the Azure portal: navigate to the resource's Settings section.

### Modifying Locked Resources
* A two-step process is required:
    * Remove the lock.
    * Perform the desired action (requires appropriate permissions).
* Resource locks override RBAC permissions. Even owners must remove the lock before making changes.
