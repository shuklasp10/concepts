# System Design Notes — Agent Instructions

> 💡 **Purpose:** Use this file as the style and structure guide for all future System Design notes in this area.

---

## Core Rules

1. **Keep the language simple and engaging.**
   - Write like you are teaching a smart junior engineer.
   - Avoid heavy jargon unless it is explained clearly.

2. **Use clear categories for major app types.**
   - Prefer 4 or 5 broad categories.
   - Good defaults: social media, e-commerce, streaming/media, productivity/collaboration, marketplace/admin.
   - If a topic needs a different category, keep it broad and practical.

3. **Examples must be category-driven.**
   - Cover each chosen category in detail.
   - For each category, include:
     - vague prompt
     - clarifying questions
     - extracted requirements
     - core flow
     - architecture hints
     - trade-offs
     - edge cases

4. **Keep examples practical.**
   - Focus on real frontend problems.
   - Show what the requirements change in the architecture.
   - Add small learning aids like quick comparison tables or memory tricks.

5. **Stay consistent across related topics.**
   - Requirement Analysis examples and Application Architecture examples should use matching category sets when possible.
   - Keep tone, depth, and structure aligned.

6. **Use the standard four-file topic layout.**
   - Every topic folder should normally contain:
     - `README.md` for the overview and navigation
     - `Concepts.md` for the core guide
     - `Examples.md` for applied walkthroughs
     - `Interview-Strategy.md` for interview prep and communication guidance
   - Keep supporting material inside those files instead of splitting into extra topic files like `Trade-Offs.md`, `Resources.md`, or `Interview-Questions.md`.

7. **Prefer clarity over cleverness.**
   - Short paragraphs.
   - Simple headings.
   - Tables where they help learning.
   - Avoid overly fancy wording.

8. **Explain the why, not just the what.**
   - Every example should show why a requirement or constraint changes the design.
   - Mention the architecture signal clearly.

9. **Teach with patterns.**
   - End sections with quick patterns, checklists, or comparison notes.
   - Help the reader remember the big idea fast.

---

## Recommended Content Shape

When writing a new note or example file, try to follow this order:

1. **Core idea**
2. **What the topic means**
3. **Main patterns or categories**
4. **Detailed examples**
5. **Trade-offs**
6. **Edge cases**
7. **Quick summary or memory trick**

---

## Style Guide

- Use a friendly teaching tone.
- Use bold text for important ideas.
- Use short bullets for clarity.
- Use tables when comparing options.
- Use examples that feel real and familiar.
- Keep the writing clean enough to skim quickly.

---

## Review Checklist Before Saving Notes

- Is the language simple?
- Are the categories broad and useful?
- Does each category include enough detail?
- Are the examples aligned with frontend system design interviews?
- Does the file feel consistent with the rest of the System Design area?
