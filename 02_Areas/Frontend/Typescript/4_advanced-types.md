# Advanced Types & Meta-Programming

## Core Idea

> Advanced TypeScript is about writing code that writes types for you. Instead of manually updating multiple interfaces when a data shape changes, you use Utility Types, Mapped Types, and Conditional Types to programmatically transform one type into another.

---

## 1. Type Operators: `keyof` and `typeof`

### `typeof` (In a Type Context)
In Vanilla JS, `typeof` is used at runtime to check if something is a `"string"` or `"object"`. 
In TypeScript, when used in a **Type Context**, it extracts the exact shape of an existing JavaScript variable and converts it into a TypeScript type.

```typescript
const defaultConfig = { theme: "dark", retries: 3 };

// TS reads the object and creates: { theme: string; retries: number; }
type Config = typeof defaultConfig; 
```

### `keyof`
It takes an object type and produces a Union Type of all its keys.
- **Mental Model:** An **Index generator**. It looks at a book (Interface) and gives you a list of all the chapters (Keys).

```typescript
interface User { id: number; name: string; email: string; }

// Result: "id" | "name" | "email"
type UserKeys = keyof User; 
```

## 2. Utility Types (The Core 5)

Utility types are built-in generic tools that transform existing types.

### `Partial<T>` and `Required<T>`
- **`Partial<T>`:** Takes a type and makes every single property optional (`?`). Great for update/patch API payloads where you only send changed fields.
- **`Required<T>`:** The exact opposite. Takes a type and strips away all `?` modifiers, forcing every field to be provided.

### `Pick<T, Keys>` and `Omit<T, Keys>`
- **`Pick<T, Keys>`:** Creates a new type by selecting a specific subset of properties.
- **`Omit<T, Keys>`:** Creates a new type by grabbing all properties *except* the ones you specify.

```typescript
interface Product { id: number; name: string; price: number; description: string; }

// I just want the name and price for the list view
type ProductCard = Pick<Product, "name" | "price">;

// I want everything EXCEPT the description
type ProductWithoutDesc = Omit<Product, "description">;
```

### `Record<Keys, Type>`
Creates an object type where you define the strict type for the Keys and the strict type for the Values.
- **Mental Model:** A **strict dictionary**.
```typescript
// An object where every key is a string, and every value is a boolean
type FeatureFlags = Record<string, boolean>;
```

## 3. Mapped Types

### Problem It Solves
What if `Partial` or `Pick` didn't exist? How would you manually write a type that makes every property in an interface optional? You'd have to rewrite it. Mapped Types allow you to loop through keys and apply transformations.

### How it Works
It uses the `in` keyword to iterate over a union of keys (usually generated by `keyof`).

```typescript
// This is exactly how Partial<T> is built under the hood!
type MyPartial<T> = {
  [Key in keyof T]?: T[Key];
};
```
- **Mental Model:** A **`for` loop for types**. It says: "For every `Key` inside the keys of `T`, create a property, make it optional (`?`), and set its value type to the original type (`T[Key]`)."

## 4. Conditional Types and `infer`

### Conditional Types
TypeScript allows you to use ternary operators (`condition ? true : false`) at the type level.

```typescript
// If T is a string, resolve to true, else resolve to false
type IsString<T> = T extends string ? true : false;
```

### The `infer` Keyword
Used *exclusively* inside conditional types. It tells TypeScript to pause, figure out a specific internal type on the fly, and assign it to a temporary variable that you can use in the `true` branch.

```typescript
// If T is an Array of *something*, capture that *something* in a variable called 'Item', and return it.
type UnpackArray<T> = T extends Array<infer Item> ? Item : T;

type A = UnpackArray<string[]>; // Resolves to 'string'
```
- **Mental Model:** A **type extractor claw**. You drop the claw (`infer U`) into a complex generic structure (like an Array or Promise), grab the hidden inner type, and pull it out for your own use.

## 5. Discriminated Unions

### Problem It Solves
When you have a Union Type containing several complex objects, TypeScript won't let you access specific properties unless it is 100% sure which object it currently is. 

### How it Works
You add a **common literal property** (the "discriminator") to every object in the union. By checking that single property in an `if` or `switch` statement, TypeScript instantly narrows down the type.

```typescript
interface Circle { kind: "circle"; radius: number; }
interface Square { kind: "square"; sideLength: number; }

type Shape = Circle | Square;

function getArea(shape: Shape) {
  // shape.radius // ERROR! It might be a square.
  
  if (shape.kind === "circle") {
    // TS knows it is 100% a Circle here. 
    return Math.PI * shape.radius ** 2;
  }
}
```
- **Mental Model:** A **name tag**. Every object wears a literal string name tag (`kind: "circle"`). When the bouncer (TypeScript) checks the name tag, it immediately knows what VIP area (properties) the object is allowed to access.

## 6. Covariance and Contravariance

This is the most academically complex topic in TypeScript, relating to how strict the compiler is when substituting types.

- **Covariance (Return Types):** You can safely return a *more specific* type than requested. If a function expects you to return an `Animal`, and you return a `Dog` (which extends `Animal`), that's perfectly safe.
- **Contravariance (Function Parameters):** You can safely pass a function that accepts a *less specific* (wider) type. If a system expects a callback `(dog: Dog) => void`, passing a callback `(animal: Animal) => void` is safe, because the callback knows how to handle *any* animal, meaning it can definitely handle a Dog.

## Interview Perspective

**Q: What is `typeof` in the TypeScript type system?**
It extracts the structural type from an existing JavaScript variable or object at compile time, saving you from manually writing out an interface that matches the object.

**Q: What is `keyof` in TypeScript?**
An operator that takes an object type and returns a Union Type consisting of all the property names (keys) of that object.

**Q: Explain Partial, Required, Pick, Omit, and Record.**
- `Partial<T>`: Makes all properties optional.
- `Required<T>`: Makes all properties required.
- `Pick<T, Keys>`: Extracts a subset of properties.
- `Omit<T, Keys>`: Removes a subset of properties.
- `Record<K, V>`: Creates an object type with specific Key and Value types.

**Q: What are utility types in TypeScript?**
Built-in generic types (like `Partial`, `Pick`, `Omit`) that globally facilitate common type transformations, preventing code duplication.

**Q: What are mapped types?**
A way to create new types by iterating over the keys of an existing type using the `in` keyword, often applying modifiers (like making them readonly or optional) during the iteration.

**Q: What are conditional types?**
Types that resolve dynamically based on a condition, using the syntax `T extends U ? TrueType : FalseType`.

**Q: What is the `infer` keyword?**
Used within Conditional Types to declare a temporary type variable that TypeScript will deduce on the fly (e.g., extracting the resolved type out of a `Promise<T>`).

**Q: What is a discriminated union?**
A pattern where multiple object types in a Union share a common literal property (like `type: 'success' | 'error'`). Checking this property allows TypeScript to narrow down the union to a specific object type safely.

**Q: Explain covariance and contravariance in TypeScript.**
They dictate type compatibility. Covariance allows a more specific type to be substituted for a generic one (common in return types). Contravariance allows a more generic type to be substituted for a specific one (common in function parameters).

---

## Key Takeaways

- Use `Pick` and `Omit` instead of duplicating interfaces for slightly different API payloads.
- Use `typeof` to grab types from config objects or third-party libraries that didn't export their types.
- **Discriminated Unions** are the absolute best way to handle complex state (like Redux actions or API response states).
