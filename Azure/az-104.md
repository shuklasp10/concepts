## Azure Resource Manager 
ARM [goto resource](https://learn.microsoft.com/en-us/training/modules/use-azure-resource-manager/2-review-benefits)

### ARM - Introduction
> Service that manages Resource groups

* __ARM__ is service that provide functionality of Resource Group and its management.
* __Basic Concept__
    * Generally all resources required by an application (VMs, VNet, Storage Account, Web App etc) are grouped as Resource Groups
    * These component related and interdependent parts of a single entity
    * ARM helps to deploy, manage, and monitor them as a group.
* A template can be used to deploy same application for different environment (test, development, production)
* ARM provides following tools to manage resource after deployment
    * Security
    * Tagging
    * Auditing

#### Consistent management layer
* provides a consistent management layer to perform tasks through     
    * Azure PowerShell
    * Azure CLI
    * Azure portal
    * REST API
    * Client SDKs

![ARM](https://learn.microsoft.com/en-us/training/wwl-azure/use-azure-resource-manager/media/resource-manager-016a1bac.png)

#### Benefits
* Group resource for deployment, management, and monitoring
* Consistent deployments throughout development lifecycle
* Infrastructure management with declarative templates instead of scripts
* Define resource dependencies for correct deployment order
* Group-level access control with RBAC integration in ARM
* Logical resource organization with tags
* Cost clarity by tagging resources

#### ARM Tips
* Declarative infrastructure definition with ARM templates (instead of imperative commands)
    * __Declarative commands:__ Create a virtual machine with 2 CPUs and 4 GB of memory.
    * __Imperative commands:__ Power on the virtual machine named 'myVM'.
* Self-contained deployment with all configuration in templates (no manual steps)
* Run imperative commands to manage resources
* Tags for additional resource organization

### ARM - Terminology
1. __Resource:__ Manageable item in Azure (e.g., VM, storage, database)
2. __Resource Group:__ Container for related Azure resources for a solution/application
3. __Resource Provider:__ Service that supplies resources for deployment and management (e.g., Microsoft.Compute for VMs)
4. __Template:__ JSON file defining resources and dependencies for repeated deployment
5. __Declarative Syntax:__ Specifying desired infrastructure state in Resource Manager templates (Here is what I intend to create)

### ARM - Resource Group
* Resources can be deployed in new or existing Resource Group
* If deployment fails for single resource, group or template. Information can be found in output
* Deployment of type of resource will not remove existing reosurce of same type

#### ARM - Resource Group - Rules
* Always Group resources with the same lifecycle (deploy, update, delete together). If different put in another Resource Group.
* Resources can be added/removed from groups at any time.
* One resource per group (like one file per folder).
* Resource groups cannot be renamed (but resources can be moved).
* Groups can contain various resource types (services) and regions.
* Use resource groups for access control to manage resources as a unit.
* Resources in different groups can interact with each other (e.g., web apps and databases).

#### ARM - Resource Group - Location
* Stores metadata about the resources in the group.
* Location determines where this data is stored (important for compliance).

#### ARM - Resource Group - Benefits
* Organize and manage resources efficiently.
* Simplified access control and permissions.
* Easier deployment, updates, and deletion of resources.
* Track resource lifecycles and interactions.

### ARM - Locks

* Lock can be applied to Subscription, Resorce Group and Resources
* Locks are inherited by child

#### ARM - Locks - Types
* __Read-only Locks:__ Prevent any change to resource

### Cloud Shell - Benefits

* __No local setup:__ Interact with Azure resources without installing tools.
* __Persistent storage:__ Save files between Cloud Shell sessions for later use.
* __Choice of shells:__ Manage Azure with either Bash or PowerShell.
* __Integrated editor:__ Edit scripts directly within the Cloud Shell interface.

### Cloud Shell - Limitations
* __Session timeout:__ Long-running activities (over 20 minutes) can lead to session disconnection and data loss.
* __No admin access:__ Cloud Shell doesn't offer sudo privileges for Azure CLI or PowerShell.
* __Limited toolset:__ Installation of unsupported tools might require a custom VM or container environment.
* __Single region storage:__ Cloud Shell's storage is region-specific.
* __No concurrent sessions:__ Only one Cloud Shell instance is allowed per user, limiting work across subscriptions or tenants.

## Azure Resource Manager Templates
> precisely defines all the ARM resources in a deployment

### ARM Templates - Introduction
* If the resource already exists and no change is detected in the properties, no action is taken. If the resource already exists and a property has changed, the resource is updated. If the resource doesn't exist, it's created.

### ARM Templates - Benefits
* __Consistent deployments:__ Templates ensure everyone uses the same language and structure for deployments.
* __Handle complex deployments:__ Templates define the correct order for deploying dependent resources.
* __Minimize errors:__ Automated deployments reduce manual configuration mistakes.
* __Infrastructure as Code:__ Templates are treated like code, allowing for version control, testing, and sharing.
* __Enable reuse:__ Templates can be parameterized for creating different environments (staging, production).
* __Modular design:__ Link smaller templates together to build complex solutions.
* __Simplified orchestration:__ Deploying a single template triggers the deployment of all your resources.


### ARM Templates - Schema
> JSON Syntax

| Element |Required|Description|
|--------|-------|-------|
| $schema | Yes | Location of the JSON schema file that describes the version of the template language. Use the URL shown in the preceding example. | 
| contentVersion | Yes | Version of the template (such as 1.0.0.0). You can provide any value for this element. Use this value to document significant changes in your template. This value can be used to make sure that the right template is being used. | 
| parameters | No | Values that are provided when deployment is executed to customize resource deployment. | 
| variable | Yes | Values that are provided when deployment is executed to customize resource deployment. | 
| function | Yes | Values that are provided when deployment is executed to customize resource deployment. | 
| resources | Yes | Resource types that are deployed or updated in a resource group. | 
| outputs | Yes | Values that are returned after deployment. | 

```
{
    "$schema": "http://schema.management.azure.com/schemas/2019-04-01/deploymentTemplate.json#",
    "contentVersion": "",
    "parameters": {},
    "variables": {},
    "functions": [],
    "resources": [],
    "outputs": {}
}
```

### ARM Templates - Parameter
> Input values to template for deployment

* parameter properties
```[JSON]
"parameters": {
    "<parameter-name>" : {
        "type" : "<type-of-parameter-value>",
        "defaultValue": "<default-value-of-parameter>",
        "allowedValues": [ "<array-of-allowed-values>" ],
        "minValue": <minimum-value-for-int>,
        "maxValue": <maximum-value-for-int>,
        "minLength": <minimum-length-for-string-or-array>,
        "maxLength": <maximum-length-for-string-or-array-parameters>,
        "metadata": {
        "description": "<description-of-the parameter>"
        }
    }
}
```
* Example of parameter
```
"parameters": {
  "adminUsername": {
    "type": "string",
    "metadata": {
      "description": "Username for the Virtual Machine."
    }
  },
  "adminPassword": {
    "type": "securestring",
    "metadata": {
      "description": "Password for the Virtual Machine."
    }
  }
}
```
#### ARM Templates - Parameter - Keypoints
In a template there can be 256 parameters.

### ARM Templates - Bicep
> Unlike JSON, Bicep is domain specific language (DSL)
* Uses declarative syntax
* concise syntax
* reliable type safety
* code reuse
* Bicep is abstraction of JSON template so include all its features
![Bicep](https://learn.microsoft.com/en-us/training/wwl-azure/configure-resources-arm-templates/media/bicep.png)
* __Transpiling__ conversion of Bicep to JSON

#### ARM Templates - Bicep - Benefits
* __Simpler & Readable:__ Bicep offers a cleaner syntax with direct parameter and variable referencing, string interpolation, and easier property access.
* __Modular Design:__ Break down complex deployments into reusable modules for better organization and management.
* __Automatic Dependency Handling:__ Bicep automatically detects resource dependencies, reducing manual effort.
* __Enhanced Authoring Experience:__ Visual Studio Code Bicep extension provides type validation and IntelliSense for a smoother writing experience.

### Microsoft Entra - AD DS vs Entra
#### AD DS
* __Structure__ hierarchical X.500-based structure
* __Resource locating__ uses DNS
* __Query and Manage protocols__ LDAP (Lightweight Directory Access Protocol)
* __Authentication__ Kerberos protocol
* __Management__  OU and GPOs
* __Trust__ uses trust between domains for delegated management.
* AD DS can be deployed in VM but it does not use Entra ID service

#### Entra
  * __Structure__ flat
  * Multi tenant directory services
  * __Query and Manage protocols__ REST Api via HTTP (Port 43) and HTTPS(port 443)
  * __Authentication__ HTTP HTTPS protocols such as
    * SAML
    * WS-Federation
    * OpenID Connect
  * __authorization__ OAuth protocol
  * __trust__ uses federation service for third party services to provide SSO

### Administrative Units
> Limits administrator's access

* Divides company into section for easy manageability
* Each sections are called Administrative Units (AU)
* restricts access of admin to AU
* AUs is managed by central admin
* Specific AU admin can manage that AU users in Entra ID
* AU admins can access other AU resources as regular users

#### AU - Implementation
* __Create an AU:__ A central admin creates an AU named "Engineering Department."
* __Populate the AU:__ Only Engineering department students, staff, and resources (e.g., servers) are added to this AU.
* __Assign the AU Role:__ An "Engineering Admin" role is created with permissions limited to manage Microsoft Entra users within the "Engineering Department" AU.
* __Assign IT Admins:__ The Engineering department's IT team members are assigned the "Engineering Admin" role with the AU scope.

#### AU - Implementation - Keypoints
* __Management Tools:__ Azure portal, PowerShell cmdlets, or Microsoft Graph.
* __Role Requirements:__ Only Global Administrators or Privileged Role Administrators can manage AUs within the Azure portal. Ensure these central authorities are in place.
* __AU Scope:__ The AU scope applies only to management permissions. AU members and admins can still browse other users, groups, or resources outside their AU using their regular user permissions.

### Subscription - Identify regions
* __Consider resource and region deployment__ Plan resources and their regions
* __Consider service support by region__ check services available in the region 
* __Consider services that don't require regions__ -Services that are managed without resources like Entra ID
* __Consider exceptions to region pairing__ Brazil and Singapore

#### Subscription - Identify regions - Regions Pairs
* __Physical isolation:__ atleast 300 miles apart
* __Platform-provided replication:__ Geo-Redundant servies
* __Region recovery order:__ : In pair outage, fixing one of region is priortized
* __Sequential updates:__ rolls in one region at a time in region pair
* __Data residency:__ Data available in same region. Except Brazil and Singapore.

### Subscription - Resource Tagging
* __Tags__
  * A tag is name value pair.
  * The tag name remains constant for all resources that have the tag applied. (doesn't change dynamically)
  * The tag value can be selected from a defined set of values, or unique for a specific resource instance.
  * A resource or resource group can have a maximum of 50 tag name/value pairs.
  * Tags applied to a resource group aren't inherited by the resources in the resource group.

## Role Based Access Control
> Who can access, What can do with resource

### RBAC - Implementation
* __RBAC__
  * Authorization system built on ARM
  * Allow an application to access all resources in a resource group.
  * Allow one user to manage VMs in a subscription, and allow another user to manage virtual networks.
  * Allow a database administrator (DBA) group to manage SQL databases in a subscription.
  * Allow a user to manage all resources in a resource group, such as VMs, websites, and subnets.

* __Core Concepts__
  * __Security Principal:__
    * Object that request for access to resource
    * User, group, service principal, managed identity
  * __Role Definition:__
    * Set of allowed operations
    * Support built in and custom definitions
    * Reader, Contributor, Owner, User Access Administrator
  * __Scope:__
    * The boundary of access, how much
    * Management group, subscription, resource group, resource
  * __Role Assignment:__
    * Attach role definition to security principal in specific scope
    * Assign the User Access Administrator role to an admin group scoped to a management group

* __Keypoints__
  * __Consider requestors:__ Internal user, external, group, App & service, Resource 
  * __Consider Roles:__ Job responsibility, Access level, Department needs
  * __Scope of permission:__ Granular control, Least Privilege Principle, Scalability
  * __Builtin vs custom:__ Customize builtin Roles

### RBAC - Create Role Definition
* __Creating Roles__
  * Set of permissions defined in JSON
  * A permission set include
    * `Actions` what actions are allowed.
    * `NotActions` what actions not allowed
    * `DataActions` what operation on data allowed
    * `AssignableScopes` list of scopes where role can be assigned
  * Example of contributor role
  ![role](https://learn.microsoft.com/en-us/training/wwl-azure/configure-role-based-access-control/media/role-definition-bf297cac.png)
  * `"*"` means all priviledges
  * NotActions:
    * `Authorization/*/Delete:` can't delete for all (all resource within the scope)
    * `Authorization/*/write:` can't change for all
    * `Authorization/elevateAccess/Action:` can't increase level or scope
* __Roles Permission__
  * Effective permission = Actions - NotActions
  * AssignableScopes -> Resource, Resource Group. Subscription, MG
  * Use action to provide broad level access and then choose notactions to narrow them down
  *  Contributer = Action(Owner) - NotActions(Delete,change,elevate)
* __Role Scopes__
  * Multiple scope
    `"subscrption/id1, subscription/id2"`
  * Only to network resource group
    `"subscription/id/ResourceGroups/Network"`
  * Available to all requestor
    `"/"`
* __Keypoints__
  * Consider using built-in roles
  * Consider creating custom definitions for business needs
  * Consider creating custom definitions
  * Consider applying deny assignments



### RBAC - Create Role Assignment
> Attaching role to requestor
* __Role Assignments__
  * By default no access is defined to requestor
  * The purpose of a role assignment is to control access.
  * Access is revoked by removing a role assignment.
  * A resource inherits role assignments from its parent resource.
  * effective permissions = Requestor Permission + Requested Resource permissions

* __Keypoints__
  * Three securtity principal supported
    * User
    * Group
    * Service Principal (App or service)

  ![role](https://learn.microsoft.com/en-us/training/wwl-azure/configure-role-based-access-control/media/role-assignment-040eb1ab.png)

### RBAC - Azure Roles vs Entra Roles
There are three types of roles for access management
1. Classic subscription administrator roles
2. Azure role-based access control (RBAC) roles
3. Microsoft Entra administrator roles

| Feature| Azure RBAC| Entra ID Admin
|--------|-----------|---------------|
| Access management	| Azure resources | Entra resources |
| Scope assignment| MG, subscriptions, resource groups, resources | Tenant level |
| Role definition | Azure portal, Azure CLI, Azure PowerShell, ARM templates, REST API | Azure admin portal, Microsoft 365 admin portal, Microsoft Graph PowerShell |
